## 1. What the Paper Did
- **Background**: Exploitation is not a binary event but a ladder of progressive capabilities, ranging from executing a buggy line of code to achieving full control over the target. Existing LLM security benchmarks treat a crash as successful exploitation, collapsing the difficult transition from triggering a bug to constructing reusable primitives and control.
- **Limitations of Prior Work**: Current benchmarks fail to distinguish between models that merely find crashes and those that can construct advanced exploitation primitives like arbitrary read/write. This simplification prevents an adequate understanding of where models fail and hinders the measurement of their improvement trajectory.

## Core Contributions
- **Proposed ExploitBench**: A capability-graded benchmark that decomposes exploitation into 16 measurable flags, covering stages from coverage and crash to sandbox primitives, arbitrary read/write, control-flow hijack, and arbitrary code execution (ACE).
- **Challenge 1: How to precisely and automatically verify intermediate exploitation steps?**
  - **Difficulty**: Binary outcomes obscure nuanced progress, and verifying memory operation primitives is prone to environmental noise.
  - **Solution**: Introduced a deterministic oracle using per-run randomized challenge-response mechanisms for primitives, differential execution against ground-truth binaries to measure progress, and signal-handler proofs for code execution, enabling fine-grained verification of each capability stage.
- **Challenge 2: How to evaluate model capabilities in realistic, heavily defended environments?**
  - **Difficulty**: Many existing benchmarks rely on simplified programs, fuzzing harnesses, or disabled defenses, failing to reflect real-world attack surfaces.
  - **Solution**: Instantiated ExploitBench on 41 V8 engine bugs, requiring agents to drive the same JavaScript/WebAssembly attack surface exposed to real attackers in identical configurations, thus assessing actual exploitation capabilities against hardened targets.

## Implementation and Evaluation
- **Experimental Setup**: Evaluated three arms: ⟨model, env⟩ (primary capability measurement), ⟨model, env, adaptive coaching⟩ (testing adaptive feedback), and ⟨model, env, CLI⟩ (ablation study on vendor-side optimizations). Tested 8 publicly deployed frontier models and one non-public research-preview model (Anthropic Mythos Preview).
- **Key Results**:
  - **Public vs. Private Models**: Public frontier models routinely reached vulnerable code and triggered crashes but failed to reliably construct primitives to escape the V8 heap sandbox. Only GPT-5.5 achieved control-flow hijack on a single WebAssembly bug; no other public model achieved ACE.
  - **Private Model Performance**: The non-public model achieved ACE on 18 out of 41 bugs in the primary arm without vendor scaffolding, demonstrating that constructing exploitation primitives is feasible within a 300-turn budget.
  - **Impact of Auxiliary Features**: Adaptive coaching and context management did not constitute capability barriers but reduced the time and budget required to achieve capabilities.
- **Comparison Conclusion**: Unlike previous benchmarks that treat exploitation as a binary outcome, ExploitBench reveals the capability gap in current public LLMs for constructing complex exploitation chains against hardened targets, establishing exploit construction as an emerging frontier capability.

#### Summary
ExploitBench addresses the oversimplification of existing LLM security benchmarks by decomposing exploitation into a fine-grained capability ladder. The results indicate that while current public top-tier LLMs can easily trigger crashes, they struggle significantly with constructing advanced exploitation primitives for arbitrary code execution, whereas private frontier models show a distinct advantage. This benchmark provides a more accurate metric for evaluating and enhancing LLM capabilities in cybersecurity.
