## 1. What the Paper Does
- **Background**: LLM-based coding agents are evolving from isolated code completion to agentic software engineering, requiring persistent memory across long sessions (e.g., prior fixes, rejected hypotheses, terminal outputs).
- **Limitations of Existing Work**: Static vector stores or generic Retrieval-Augmented Generation (RAG) are insufficient for Reinforcement Learning (RL) code development. In RL contexts, minor implementation details can alter Bellman targets, terminal masks, gradient flows, or validation claims, which generic RAG fails to handle with necessary precision and auditability.

## Core Contributions
- **Proposed**: RL Developer Memory, a local-first, Model Context Protocol (MCP)-native developer-memory architecture designed specifically for RL coding agents.
- **Challenge 1: Precision and Auditability of Memory Selection**. The difficulty lies in treating memory retrieval as a logged contextual decision process rather than simple similarity matching. The paper addresses this by using `issue_match` to rank candidates and log telemetry, `issue_feedback` to map raw labels to bounded canonical rewards, and `issue_record_resolution` to link verified resolutions back to earlier retrieval events.
- **Challenge 2: Balancing Safe Deployment with Policy Learning**. The challenge is introducing learned policies without compromising system stability. The solution involves keeping a deterministic control path as the deployed policy, while running a diagonal contextual-bandit residual policy in "shadow mode," affecting canary behavior only through conservative Off-Policy Evaluation (OPE) gates.

## Implementation and Evaluation
- **Implementation & Deployment**: The system includes theory-to-code metadata, validation-tier promotion checks, and review-gated governance. It currently does not support active learned-policy deployment or official-client MCP interoperability.
- **Evaluation Results**: Evaluated on a deterministic 200-case benchmark covering RL algorithm bugs, hard negatives, etc.
    - **Accuracy**: Both deterministic control and full shadow/OPE configurations achieved 80.0% expected-decision accuracy.
    - **Hard-Negative Suppression**: Both achieved 100.0% suppression.
    - **Other Metrics**: Passed 11/11 static validation checks and 10/10 dynamic integration cases. The full configuration added learning telemetry rather than demonstrating an accuracy gain over the deterministic baseline.
- **Comparison & Limitations**: Compared to generic RAG, this architecture offers auditable memory control. Limitations include p95 latency regression in live full configurations and 40 residual failures in non-RL path, scope, and alias compatibility cases.

## 4. Conclusion
The paper contributes an auditable memory-control architecture with component-level evidence and explicit claim boundaries, rather than claiming a universal improvement for coding agents. It emphasizes that memory management in RL coding agents requires strict governance, offline evaluation, and safety gating to handle the significant impact of minor details on training stability and correctness.
